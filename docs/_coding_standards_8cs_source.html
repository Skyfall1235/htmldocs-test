178 //4. Code Design Principles
179 //4.1 Methods
180 //Single Responsibility: One responsibility per method.
181 //Descriptive Names: Be specific, use verbs.
182 //Small Size: Fit in one screen.
183 //Input Validation: Especially for external input.
184 //Avoid Side Effects: Be explicit about state changes.
185 //Return Values: Return data, don’t modify external variables.
186 //Favor Immutability: Avoid modifying inputs directly.
187
188 //4.2 Classes
189 //Single Responsibility: One reason to change.
190 //Cohesion: Related members grouped logically.
191 //Low Coupling: Use events, not direct calls.
192 //Encapsulation: Use access modifiers properly.
193 //Favor Composition Over Inheritance
194
195 //SOLID Principles:
196 //S: Single Responsibility
197 //O: Open/Closed
198 //L: Liskov Substitution
199 //I: Interface Segregation
200 //D: Dependency Inversion
201
202 //4.3 Interfaces
203 //Contract Definition: Promote consistency.
204 //Loose Coupling: Depend on abstractions.
205 //Polymorphism Support: Use interface types.
206 //Testability: Use mocks/stubs.
207 //Role-Based Design: Interfaces define behavior.
208 //Example:
209 public interface IAttackable
210 {
211 void TakeDamage(int damage);
212 }
213 //4.4 Scriptable Objects
214 //Data Containers: Store independent data.
215 //Reusability: Shared across GameObjects.
216 //Inspector Customization: Use custom editors.
217 //Use Cases: Game settings, shared data, configs.
218
219 //5. Good Practices
220 //5.1 Assembly Definitions
221 //Modular Code: Organize logically.
222 //Compilation Time: Reduce recompilation.
223 //Dependency Management: Define dependencies.
224 //Folder Structure: Match assembly layout.
225
226 //5.2 Folder Structure
227 //Use a logical, modular layout:
228
230 // /Core
231 // /GameSystems
232 // /Managers
233 // /Gameplay
234 // /Player
235 // /Enemies
236 // /AI
237 // /UI
238 // /Data // ScriptableObjects
239 // /Editor // Editor scripts
240
241 //5.3 Avoid Deeply Nested Prefabs
242 //Keep prefabs simple and shallow.
243 //Break into smaller prefabs and compose at runtime.
244
245 //5.4 Use Object Pooling
246 //Reuse objects (e.g., bullets, enemies).
247 //Reduces GC and improves performance.
248
249 //5.5 Optimize Asset Usage
250 //Use proper formats, compression.
251 //Consider Asset Bundles or Addressables.
