7 // Coding Standard: Git Commit Message Guidelines
8 //-----------------------------------------------------------------------------
9 //
10 // This document outlines the standard for writing clear, consistent, and traceable
11 // Git commit messages within our project. Adhering to these guidelines improves
12 // collaboration, code review efficiency, and project history understanding.
13 //
14 //-----------------------------------------------------------------------------
15 // I. Commit Summary (Subject Line)
16 //-----------------------------------------------------------------------------
17 //
18 // The first line of every commit message acts as its summary. It must be concise
19 // and immediately convey the change's purpose.
20 //
21 // Format:
22 // <type>(<scope>): <summary>
23 //
24 // Guidelines:
25 //
26 // 1. <type> (Required):
27 // Categorizes the nature of the change. Use lowercase.
28 // - feat : A new feature.
29 // - fix : A bug fix.
30 // - docs : Documentation-only changes.
31 // - style : Code formatting/style changes (no meaning change).
32 // - refactor: Code restructuring (neither fixes a bug nor adds a feature).
33 // - perf : Performance improvement.
34 // - test : Adding or correcting tests.
35 // - build : Changes to the build system or external dependencies.
36 // - ci : Changes to CI configuration files and scripts.
37 // - chore : Miscellaneous or routine maintenance tasks.
38 // - revert : Reverts a previous commit.
39 //
40 // 2. (<scope>) (Optional):
41 // Indicates the specific part of the codebase or module affected. Use lowercase.
42 // Examples: (auth), (ui), (api), (database).
43 //
44 // 3. <summary> (Required):
45 // A concise, imperative (e.g., "Add", "Fix"), and capitalized description.
46 // - Start with a capital letter.
47 // - Do NOT end with a period.
48 // - Keep under 72 characters for readability.
49 // - Focus on *what* was changed.
50 //
51 // Example Summary:
52 // feat(checkout): Implement guest checkout flow
53 //
54 //-----------------------------------------------------------------------------
55 // II. Commit Description (Body)
56 //-----------------------------------------------------------------------------
57 //
58 // After a blank line following the summary, the description body details the
59 // files modified, added, or deleted by the commit. This provides quick insight
60 // into the change's scope.
61 //
62 // Guidelines:
63 //
64 // 1. Separation:
65 // Leave a blank line between the summary and the description.
66 // 2. File List:
67 // Use bullet points to list relevant files affected by the commit.
68 // Use relative paths from the repository root.
69 // 3. Issue References:
70 // Always reference relevant issue tracker IDs (e.g., "Closes #123", "Fixes BUG-456").
71 //
72 // Example Description:
73 //
74 // - src/features/checkout/components/GuestCheckoutForm.tsx
75 // - src/features/checkout/services/orderService.cs
76 // - src/routes/AppRoutes.cs
77 //
78 // Closes #SUP-456, Fixes BUG-456
79 //
80}
